Nested improves query correctness, but increases:
    index size  [one nested document is stored as a separate document in the index]
    query complexity [Nested queries are more complex than regular queries, as they require additional processing to join the nested documents with their parent documents.]
    update cost [Updating a nested document requires updating the entire parent document, which can be costly in terms of performance and resources.]

When should we avoid nested?
    When arrays are large [Nested documents can significantly increase the index size and query complexity, so if the arrays are large, it may be better to use a different approach, such as denormalization or parent-child relationships.]
    When data is frequently updated [Since updating a nested document requires updating the entire parent document, it can be costly in terms of performance and resources. If the data is frequently updated, it may be better to use a different approach, such as denormalization or parent-child relationships.]
    When object bondaries are not important [If the fields in the array do not need to stay together, and you do not require accurate queries or aggregations on those fields, then using a nested type may not be necessary. In such cases, you can use a regular object type, which is more efficient in terms of index size and query performance.]

        Limitation of denormalization:
            A product manager asks:
                Return all movies that have at least one action rating,
                but sort them by their highest rating overall, regardless of genre.

            This is not possible as "Return all movies that have at least one action rating" will require us to filter by rating.genre: action,
            which will exclude non-action ratings from the results.
            As a result, we cannot sort by the highest rating overall, regardless of genre, because we have already filtered out non-action ratings.


Nested vs parent-child tradeoffs
    Nested → best for small related arrays
    Parent–child → best for very large, frequently updated child datasets

Debugging incorrect nested matches
    1. Check mapping → is it nested?
    2. Check query → wrapped in nested?
    3. Check sorting → nested path specified?
    4. Check aggregations → nested aggregation used?
